AI Collaboration Log - Architecture, Debugging, and Product Thinking

One of the most useful things about working with AI during this project was being able to use it as a real sounding board, not just a code generator. A good example was the early architecture debate around whether to use Next.js as a full-stack framework or go with a FastAPI backend and a separate React frontend. The AI made a fair case for Next.js since it would cut down on boilerplate and deployment complexity. But I pushed back, and here is why: I want a native Python environment on the backend. The whole future of this product is a personalized recommendation engine using PCA for user segmentation and RAG for grounding the AI insights in real news. That pipeline lives in Python, in scikit-learn and sentence-transformers, not in a Next.js API route. By choosing FastAPI plus React as two separate services, I enforced a clean contract between the frontend and the data layer from day one. That is a real production architecture, not a prototype shortcut.

The debugging sessions were also where I felt the AI collaboration was most valuable, because it forced me to actually understand what was going wrong rather than just patch symptoms. One issue that took real investigation was a silent 500 error on the auth endpoints. The app would start fine, the register request would reach the server, but the response was always an internal error with no helpful message. I traced it back to the JWT_SECRET environment variable not being set locally. When the secret is missing the jose library fails to sign the token, but the error was not surfacing cleanly. The fix was straightforward once I understood the cause, but finding it required reading through the auth module line by line and thinking about what could fail silently at startup versus at request time. I wrote it up and made the error handling more explicit so it would never be ambiguous again.

The other debugging moment worth mentioning is the database schema mismatch. SQLAlchemy's create_all function is great for greenfield setups but it does not alter existing tables. So when I added the name and hashed_pw columns to the User model, nothing broke on startup and nothing was migrated either. The columns simply did not exist and every register attempt was failing at the INSERT. I spotted this by inspecting the live table columns directly through the DB client, and then wrote a reset_db script to tear down and recreate the schema cleanly. That is a classic ORM footgun and it is something that would trip up anyone who has not run into it before.

On the product side, the meme categorization feature is something I thought about carefully. The Imgflip API gives us legal, royalty-free meme templates, which is important for a deployed product. But the raw binary vote, thumbs up or thumbs down on a meme, is not that useful on its own for building a recommendation model. What matters is why someone liked it. By running a lightweight keyword categorization over the meme template name, I can map votes to archetypes: someone who consistently upvotes bull market memes is a very different user than someone who upvotes bear market pain memes. Those personas correspond to real behavioral differences in how people want to consume crypto content. A HODLer who loves rocket-to-the-moon humor wants very different insights than someone who enjoys commiserating about a crash. So the category field on each vote is not just a label, it is the first step toward building a meaningful user preference vector that the future recommendation model can actually use.
