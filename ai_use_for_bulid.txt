AI Collaboration, Product Strategy & Architectural Thought Process

Throughout the development of this AI Crypto Advisor, I utilized AI (Claude/Gemini) primarily as an architectural sounding board, a pair-programmer for edge-case debugging, and a tool to accelerate boilerplate generation. My overarching goal was to ensure the system's structural integrity, maintain clean code standards, and build a scalable product.

Below is a deep dive into my thought process and how AI was integrated into the development lifecycle:

1. Product Discovery & Overcoming Domain Gaps

Full Transparency: I do not come from the crypto space, and my initial knowledge of this domain was minimal. This is exactly why my first step was not opening the IDE, but rather initiating a deep-dive product discovery session to understand the target users and their pain points.

I analyzed the target audience (crypto investors ranging from HODLers to Day Traders) and their primary pain point: information overload. I concluded that the value proposition of this dashboard is not just showing raw market data, but curating it. The AI Insight feature and the dynamic onboarding quiz were designed based on this reasoning — the dashboard dynamically adapts to the user's specific persona, mixing hard financial data (CoinGecko) with behavioral sentiment and humor (Reddit Memes).

AI was used here as a discussion partner to pressure-test ideas, not to generate them. The product decisions were mine.

---

2. Architectural Paradigm: Why Decoupling Matters

When deciding on the tech stack, I evaluated full-stack frameworks (like Next.js) versus a decoupled approach. I explicitly chose to build a decoupled system: React (Vite) for the frontend and FastAPI (Python) for the backend.

Separation of Concerns: The client is purely representational; the server handles all business logic, authentication, and external API orchestration. This enforces strict SoC and SOLID principles from day one.

ML Future-Proofing: Since the assignment includes a bonus requirement around Machine Learning (see BONUS_ML_PLAN.md), writing the backend in Python from the start was a strategic decision I made — not one suggested by AI. Python provides a native, frictionless ecosystem for data science libraries (Pandas, Scikit-Learn, PyTorch), allowing a seamless transition from a standard CRUD backend to a recommendation engine. A Next.js API route would have made that future work unnecessarily painful.

---

3. Database Modeling & ML Recommendation Infrastructure

Designing the database schema to support the future voting and recommendation system was a phase where I had to think carefully about trade-offs.

The Challenge: How do we store user feedback (thumbs up/down) in a way that is structured enough for ML algorithms, yet flexible enough to handle different content types (memes, news, coins)?

My Solution: I implemented a normalized votes table that captures content_type, content_key, category (crucial for feature extraction), and vote_value. The category field was my design decision — it is what allows the system to eventually build per-user preference vectors (e.g., a user who consistently upvotes bull_market memes vs. one who upvotes bear_market content are meaningfully different users). AI did not suggest this schema; I designed it with the ML pipeline in mind.

For user profile data (onboarding preferences), I used a flexible JSON column instead of rigid SQL columns, which avoids painful schema migrations as requirements evolve.

---

4. Frontend Engineering & UI/UX Philosophy

I strongly believe clean code applies to the UI as much as the backend. I refactored the React codebase to follow DRY and component-driven principles.

Instead of duplicating voting logic across four different cards, I extracted it into a reusable useVoting custom hook and a shared ThumbsVote component. I designed this abstraction — AI helped translate the design into working JSX.

Design System: I directed the AI to implement a cohesive "Soft UI" design system using Tailwind CSS (glassmorphism, subtle gradients, fixed-aspect ratios) to prevent layout shifts and create a fintech-grade feel. I specified the visual language (indigo/violet palette, slate dark mode, rounded-3xl cards); AI produced the Tailwind class combinations.

---

5. API Integration — Bugs I Found and Fixed Myself

This is where I diverged from AI most significantly. AI does not have up-to-date knowledge of third-party API responses, so I had to debug these integrations myself:

Reddit meme API (meme-api.com): The API sometimes returns video posts (.mp4 URLs) without setting the isVideo flag. AI's first version only checked the isVideo flag, so videos were being sent to the frontend as image src attributes and the image tag was silently broken. I caught this by testing with live API responses, traced the cause, and added an additional .endswith(".mp4") check to the url_is_video condition.

OpenRouter AI model: AI suggested model identifiers that no longer exist on the free tier. I went to the OpenRouter documentation, identified the currently available free model, and updated the config. The AI had no way to know this had changed since its training cutoff.

CORS configuration: The Vite dev server runs on port 5173 and the FastAPI backend on port 8000. The initial setup had no CORS middleware, so all frontend requests were blocked. I read the FastAPI CORS docs and configured the CORSMiddleware with the correct origins for both development and production.

News RSS fallback chain: The CoinTelegraph and Decrypt RSS feeds have different XML structures and date formats (RFC 2822). I wrote the date parsing function and field-path logic myself after inspecting real feed responses — AI had the structure wrong on the first pass.

6. DevOps & Deployment Strategy

When approaching deployment, I mapped out the infrastructure for a decoupled stack using free-tier services. After researching the options, I chose the following split:

Neon.tech (Database): A serverless PostgreSQL provider. I chose Neon specifically because it offers a full PostgreSQL instance on the free tier with no credit card required, it is fully compatible with the PostgreSQL dialect we use via SQLAlchemy, and its serverless architecture means zero idle cost. This was my decision after evaluating Supabase (also good but more opinionated) and Render's managed Postgres (requires a paid plan after 90 days).

Railway (Backend): Chosen for its straightforward Python/FastAPI deployment and reliable free tier. Environment variables (DATABASE_URL, OPENROUTER_API_KEY) are injected securely at the platform level, never stored in the repository.

Vercel (Frontend): Chosen for its global CDN, native Vite/React build support, and instant deployments on every git push. The frontend environment variable VITE_API_URL is set in the Vercel dashboard to point at the Railway backend domain.

This three-service split means the database, backend, and frontend are independently deployed and can be swapped without touching the others — a real production separation of concerns.

---

Summary

AI was not used to "do the work for me." It was used to accelerate boilerplate, challenge my assumptions, and act as a fast-feedback loop on implementation details. Every architectural decision — the decoupled stack, the votes schema, the ML data collection strategy, the deployment split — was made by me after weighing alternatives. The API debugging and integration fixes were entirely my own work, since AI lacked current knowledge of the specific third-party services involved.
